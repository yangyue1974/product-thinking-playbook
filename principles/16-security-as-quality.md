# 16 · 安全 = 产品品质

> **安全漏洞不是"以后的事"，是当下的产品问题。用户的账号、数据、会话一旦出问题，产品信任就崩了。**

## 核心思想

很多独立开发者把安全放在"可选项"—— **"先上线再说，有用户再补"**。这是错的：

1. 安全 bug 和功能 bug 的**修复成本**一样（都是改代码）
2. 安全 bug 的**事故代价**远大于功能 bug（一次泄漏 = 产品信任归零）
3. 上线后再补，**用户数据已经暴露在风险里**

**把安全 bug 和功能 bug 放在同一个修复队列里对待，而不是区分优先级。**

**主动审计**比被动修复重要。发现问题 ≠ 没问题。上线后的安全感，来自于"我检查过"，而不是"我觉得没问题"。把审计当成发布前的标准流程，而不是可选项。

## 在真实项目里长什么样

**Spybook** · 代码审计发现 14 个问题，涵盖安全（timing attack、open redirect、error message 泄漏）、功能（callbackUrl 丢失、SQL 注入风险）、性能（O(n) 查询）。

没有讨论"哪些值得修"，没有说"先上线再说"：

> *"全部都要修，你按计划步骤做吧。"*

**Xianxia**：

> *"自动审核整个程序并修复 bug。"*

在完成一个阶段后主动要求全面 code audit，而不是等问题暴露。

## 反模式

- **"小产品不会被攻击"**：攻击大多是自动扫描，不挑产品大小。
- **"等有用户再补"**：有用户之后补，就是用"用户的账户风险"换你的时间。
- **Exception message 泄漏**：把错误堆栈直接返回前端，把数据库结构送给潜在攻击者。
- **硬编码的回跳地址**：open redirect 隐患，默认所有回跳都接受。
- **没有 rate limit**：登录接口可以无限尝试密码。

## 思考公式

> **安全 bug 和功能 bug 在我这里是同一个队列。**  
> **上线前主动审计，不等被动修复。**

**发布前 checklist（最少版）**：
1. 认证相关：密码比对用 timing-safe、会话管理正确
2. 注入相关：SQL 参数化、用户输入不直出
3. 跳转相关：open redirect 防护
4. 错误相关：错误信息不泄漏内部结构
5. Rate limit：关键接口（登录、API 调用）有限流
6. 敏感数据：日志里不打印密码 / token
