# Contra Code · 产品思维案例

> VS Code / Cursor 内嵌魂斗罗浮层，解决程序员等待编译时的碎片化时间。  
> 从 Contra Code 项目全过程对话提炼 · 2026-04-13 至 2026-04-14

---

## 一、场景先于功能

> *"我们在 Coding 的时候都有大量的等待时间非常无聊，这个场景是一个真实的需求。"*

好产品从观察真实场景开始，不是从功能列表开始。看到别人做超级玛丽浮层，没停留在"好酷"，而是立刻意识到这是个被普遍低估的真实场景。所有后续决策都回到这个场景约束来判断：

- 玩家不会长时间投入 → 去掉关卡 / 武器升级
- 玩家要随时回到工作 → 死了就地重生
- 画面必须不干扰编程 → 底部浮层
- 按键不能和 IDE 冲突 → 焦点切换机制

**原则：** 不问"这个功能好不好"，问"这个场景到底是什么样的"。

---

## 二、借力认知，降低上手成本

> "高度致敬魂斗罗，不是 1:1 复制。"

用户已有的记忆和经验是免费资产。选择魂斗罗致敬而不是自己发明玩法——"大家一看就知道是什么"；保留情怀红利、规避版权风险；操作沿用原版（方向键瞄准、空格跳、Z 射击），老玩家零学习成本。

**原则：** 能蹭的记忆就蹭，创新留给用户看不到但感受得到的地方。

---

## 三、愿景不因实现难度降级

> *"这和我想的不一样，跟他们在旁边开一个挖地雷有什么区别。"*

我把游戏做成了 Tab，用户直接反击。这句话分量很足：原始意图是"不打断编程"，Tab 模式让人能玩，但损失了核心价值主张——用户没有因"技术上做不到"接受降级。驱动我继续研究 DOM 注入，最后落地在底部面板（技术上有妥协，但保留了"不占用编辑器区域"的核心意图）。

**原则：** "能用"≠"达到目标"，愿景是议价底线。

---

## 四、让场景做减法，而不是让欲望做加法

定义玩法时主动做减法：

| 去掉 | 原因 |
|---|---|
| 武器升级 | 码农不可能超时间玩 |
| 关卡通关 | 场景不适合 |
| 复杂剧情 | 玩家随时被拉回工作 |
| 用户注册 | 增加使用门槛 |

| 保留 | 原因 |
|---|---|
| 积分挑战 | 简单 |
| 就地重生 | 降低挫败感 |
| 匿名排行 | 社交但无门槛 |

用"场景"反推"哪些功能不要"，而不是用"功能列表"正推"做什么"。

**原则：** 每加一个功能，先问：这个场景下用户有这个耐心吗？

---

## 五、决策要果断，细节要挑剔

**果断决策的时刻：**
- "Contra Code 这名字太好了" —— 一秒定板
- 技术方案选一："1" —— 不纠结
- 商业方向："用 vscode 和 cursor 的码农最多，先做这两个，其他先不管"

**挑剔细节的时刻：**
- 看到粗糙美术："我们不能做一个这么粗糙的东西吧"
- Boss 太难："大家就没信心打下去了"
- 平台设计："没有什么章法，要梳理一下"
- 背景接缝、阴影线、抠图质量逐一发现并指出

**原则：** 方向问题拍板要快，体验问题绝不放过。

---

## 六、用户心理 > 行业惯例

> *"传统游戏都是用过关和通关来作为自我挑战，但是我们这个场景不适合，我觉得用积分更简单，往前跑，过关斩将，直到死，得了多少分，下次就是继续挑战这个分数。"*

定义挑战机制时的三步推理：识别行业通用做法（过关通关）→ 判断场景不适配（玩家时间碎片化）→ 推导新机制（积分 + 自我挑战）。

难度设计同理：

> *"Boss 难度太大，影响大家玩的心情…… 还是要把难度降低一点，这样玩的时间长。"*

始终站在玩家的**心情**和**持续意愿**角度做决定，不照着魂斗罗原版设计。

**原则：** 问"我的用户会怎么想"，不问"游戏应该怎么做"。

---

## 七、遇到工具瓶颈，亲自动手

- 代码抠图效果不好 → 直接自己抠好 PNG 覆盖
- MJ 生成背景不对色 → 重画并指定排除词（`--no purple, blue night`）
- 背景接缝明显 → 主动提议"画的时候要求能衔接"，给出解决方向

没有因为"这个工具不行"就放弃，而是**用结果倒推方法**。

**原则：** 别问工具能做什么，问目标要什么，工具不够就人补上。

---

## 八、视觉先沟通，避免语言歧义

- 我提供 4 种游戏位置选项时，你没看完就用语言精确描述了"底部浮层 + 半透明"的设想
- 后续所有视觉迭代用"你说 + 我画 + 你反馈"的快循环
- MJ 提示词工作流：你提要求，我写专业提示词，你执行画图——角色分工清晰

**原则：** 视觉决策需要看图讨论，不要陷入语言描述的歧义。

---

## 九、先用再评，避免想象中的需求

每一轮打磨都遵循这个模式：上线新版本 → 真玩一会儿 → 反馈**具体**问题（经常一次好几个）。

> *"人物和物品没抠图，必须抠图，火苗都抠图了，是正常的，枪械包似乎是半透明，有个透明框。"*

玩过才能清楚看到三个不同的问题。不会"担心可能有问题"就提前纠结，而是真正用过之后再列问题。

**原则：** 真实体验比想象中的问题清单更有价值。

---

## 十、闭环思维：从做出来到用得上

功能稳定后主动提出：

> *"现在我们来解决如何让用户最简单安装的问题。"*

时机很关键——不是功能全做完才想起来，也不是一开始就纠结，而是在 **MVP 稳定、准备推广时**主动思考分发。然后进一步问："商店发布和本地安装哪个简单？"——从用户视角评估分发成本。

**原则：** 做完 ≠ 做好，做好 ≠ 用得到。

---

## 被验证的几个直觉

1. **从自己的痛点出发更靠谱** —— 因为自己就是用户，反馈真实
2. **不纠结小路线差异** —— 两个选项差不多时，果断选一个试
3. **美术质量是门面** —— 功能再好，画面粗糙用户一眼就走
4. **有些东西看了才知道** —— 别在脑子里过度推演，上线最快反馈
5. **用户不关心技术难度** —— "为什么做不到"是我的问题，不是他们的
