# 我的产品思维手册
> 基于 Contra Code（VS Code 游戏插件）项目完整对话提炼

---

## 1. 场景先于功能

### 核心原则
**好产品从观察真实场景开始，不是从功能列表开始。**

### 这个项目的体现
- 起点一句话：**"我们在 Coding 的时候都有大量的等待时间非常无聊，这个场景是一个真实的需求"**
- 看到别人做超级玛丽浮层，没停在"好酷"的层面，而是立刻意识到这是个被普遍低估的真实需求
- 所有后续决策都回到这个场景约束来判断：
  - 玩家不会长时间投入 → 去掉关卡/武器升级
  - 玩家要随时回到工作 → 死了就地重生
  - 画面必须不干扰编程 → 底部浮层
  - 按键不能和 IDE 冲突 → 焦点切换机制

### 思考公式
> 不问"这个功能好不好"，问"这个场景到底是什么样的"

---

## 2. 借力认知，降低上手成本

### 核心原则
**用户已有的记忆和经验是免费资产。**

### 这个项目的体现
- 不自己发明玩法，选择 **魂斗罗致敬**："大家一看就知道是什么"
- 进一步细化："高度致敬，不是 1:1 复制"—— 保留情怀红利，规避版权风险
- 操作沿用原版（方向键瞄准、空格跳、Z 射击）—— 老玩家零学习成本

### 思考公式
> 能蹭的记忆就蹭，创新留给用户看不到但感受得到的地方

---

## 3. 原始意图不因实现难度降级

### 核心原则
**实现方案可以妥协，产品愿景不能让步。**

### 这个项目的体现
我把游戏做成了 Tab，你直接反击：

> "这和我想的不一样，跟他们在旁边开一个挖地雷有什么区别"

这句话分量很足：
- 你清楚原始意图是"不打断编程"
- Tab 模式让你能玩，但损失了核心价值主张
- 你没有因"技术上做不到"接受降级

驱动我继续研究 DOM 注入 → 最后落地在底部面板（妥协方案，但保留了"不占用编辑器区域"的核心意图）。

### 思考公式
> "能用"≠"达到目标"，愿景是议价底线

---

## 4. 让场景做减法，而不是让欲望做加法

### 核心原则
**功能不是加得越多越好，而是越贴合场景越好。**

### 这个项目的体现
定义玩法时主动做减法：

| 去掉 | 原因 |
|---|---|
| 武器升级 | 码农不可能超时间玩 |
| 关卡通关 | 场景不适合 |
| 复杂剧情 | 玩家随时被拉回工作 |
| 用户注册 | 增加使用门槛 |

| 保留 | 原因 |
|---|---|
| 积分挑战 | 简单 |
| 就地重生 | 降低挫败感 |
| 匿名排行 | 社交但无门槛 |

你用"场景"反推"哪些功能不要"，而不是用"功能列表"正推"做什么"。

### 思考公式
> 每加一个功能，先问：这个场景下用户有这个耐心吗？

---

## 5. 决策要果断，细节要挑剔

### 核心原则
**大方向快速定板，小细节反复打磨。**

### 这个项目的体现

**果断决策的时刻：**
- "Contra Code 这名字太好了" — 一秒定板
- 技术方案选一："1" — 不纠结
- 商业方向："用 vscode 和 cursor 的码农最多，先做这两个，其他先不管"

**挑剔细节的时刻：**
- 看到粗糙美术："我们不能做一个这么粗糙的东西吧"
- Boss 太难："大家就没信心打下去了"
- 平台设计："没有什么章法，要梳理一下"
- 背景接缝、阴影线、抠图质量逐一发现并指出

### 思考公式
> 方向问题拍板要快，体验问题绝不放过

---

## 6. 用户心理 > 行业惯例

### 核心原则
**别人怎么做只是参考，你用户想要什么才是答案。**

### 这个项目的体现

定义挑战机制时的三步推理：

> "传统游戏都是用过关和通关来作为自我挑战，但是我们这个场景不适合，我觉得用积分更简单，往前跑，过关斩将，直到死，得了多少分，下次就是继续挑战这个分数"

1. 识别行业通用做法（过关通关）
2. 判断场景不适配（玩家时间碎片化）
3. 推导新机制（积分 + 自我挑战）

难度设计同理：

> "Boss 难度太大，影响大家玩的心情... 还是要把难度降低一点，这样玩的时间长"

你始终站在玩家的**心情**和**持续意愿**角度做决定，不是照着魂斗罗原版设计。

### 思考公式
> 问"我的用户会怎么想"，不问"游戏应该怎么做"

---

## 7. 遇到工具瓶颈，亲自动手

### 核心原则
**工具有局限时，自己绕过去，不被流程卡住。**

### 这个项目的体现
- 代码抠图效果不好 → 你直接自己抠好 PNG 覆盖
- MJ 生成背景不对色 → 重画并指定排除词（`--no purple, blue night`）
- 背景接缝明显 → 主动提议"画的时候要求能衔接"，给出解决方向

你没有说"这个工具不行"就放弃，而是**用结果倒推方法**。

### 思考公式
> 别问工具能做什么，问目标要什么，工具不够就人补上

---

## 8. 视觉先沟通，避免语言歧义

### 核心原则
**能画出来的概念，不要只用语言描述。**

### 这个项目的体现
- 我提供 4 种游戏位置选项时，你没看完就用语言精确描述了"底部浮层 + 半透明"的设想
- 后续所有视觉迭代用 **"你说 + 我画 + 你反馈"** 的快循环
- MJ 提示词工作流：你提要求，我写专业提示词，你执行画图 —— 角色分工清晰

### 思考公式
> 视觉决策需要看图讨论，不要陷入语言描述的歧义

---

## 9. 先用再评，避免想象中的需求

### 核心原则
**亲自体验完再定问题清单。**

### 这个项目的体现
每一轮打磨都遵循这个模式：
1. 我上线新版本
2. 你真玩一会儿
3. 你反馈**具体**问题（经常一次好几个）

你不会"担心可能有问题"就提前纠结，而是真正用过之后再列问题。

**例子：**
- "人物和物品没抠图，必须抠图，火苗都抠图了，是正常的，枪械包似乎是半透明，有个透明框" —— 玩过才能清楚看到三个不同的问题
- "sentry turret 好像没出现过，是后面才会有吗，我还没打到" —— 亲自体验后发现触发门槛太高

### 思考公式
> 真实体验比想象中的问题清单更有价值

---

## 10. 闭环思维：从做出来到用得上

### 核心原则
**产品不是"能运行"就结束，"用户能用上"才是完成。**

### 这个项目的体现

功能稳定后你主动提出：

> "现在我们来解决如何让用户最简单安装的问题"

这个问题的时机很关键：
- 不是功能全做完才想起来
- 也不是一开始就纠结
- 而是在 **MVP 稳定、准备推广时** 主动思考分发

然后进一步问："商店发布和本地安装哪个简单？"—— 从用户视角评估分发成本。

### 思考公式
> 做完 ≠ 做好，做好 ≠ 用得到

---

## 总结：四象限心法

| 象限 | 心法 |
|:---:|:---|
| **场景** | 从真实场景出发，让场景做减法 |
| **借力** | 蹭得上的认知就蹭，借经典降低门槛 |
| **坚持** | 愿景不退让，细节不放过 |
| **闭环** | 从体验到分发，整条链都是产品 |

### 一句话版本

> **先找场景，借力记忆，做减法不做加法，核心愿景不妥协，细节持续打磨，做完之后还要让用户用得上。**

---

## 附：我在这个项目里被验证的几个直觉

1. **从自己的痛点出发更靠谱** — 因为自己就是用户，反馈真实
2. **不纠结小路线差异** — 两个选项差不多时，果断选一个试
3. **美术质量是门面** — 功能再好，画面粗糙用户一眼就走
4. **有些东西看了才知道** — 别在脑子里过度推演，上线最快反馈
5. **用户不关心技术难度** — "为什么做不到"是我的问题，不是他们的
